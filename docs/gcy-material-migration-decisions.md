# GCY Material Migration Decisions

> 状态：这是非常不完善的迁移草稿，只记录当前已经讨论过或临时归档的处理方式。
> 具体实现前仍需要按材料、配方、机器和 GTCEu Modern 当前源码逐项复核。

本文档记录实际迁移时已经选定的处理方式。差异来源见
`docs/gcy-material-dedupe.md` 和
`docs/gcy-duplicate-material-differences.md`。

核相关内容继续搁置；本文件只记录当前非核材料的处理决策。

## 0.0. 全量材料分包

全量材料清单的迁移分包记录在 `docs/gcy-material-dedupe.csv` 的
`migration_package` 列，并生成完整索引 `docs/gcy-material-packages.md`。

当前分包是实现路由分类，不是完整化学分类。后续源码注册时优先按这些 package 拆模块：

- `chemistry/biotech`
- `chemistry/catalysts`
- `chemistry/dyes_pigments`
- `chemistry/energetics`
- `chemistry/inorganic`
- `chemistry/organic`
- `chemistry/petrochem_fuels`
- `chemistry/polymers`
- `deferred/nuclear`
- `electronics/optical_materials`
- `elements/direct`
- `fluids/cryogenics`
- `high_energy/particles_and_exotics`
- `materials/gems`
- `metallurgy/alloys`
- `metallurgy/ore_processing`
- `voltage/superconductors`
- `voltage/tier_markers`

包表只保留 `status != exists_in_gtceu` 的材料；已经由 GTCEu Modern 提供的材料只保留在
总 CSV 和 id map 中。当前这一轮按材料名复核分类，名字本身是普通盐、氧化物、氢氧化物、
有机/有机金属试剂、燃料、聚合物或矿处理残渣时，优先按这些名字语义调整分包。

`deferred/nuclear` 只作为分类和迁移暂停标记，不表示这些材料已经进入实现范围。

## 0. 重叠材料 id 规则

适用范围：

- GCY 原版和 GTCEu Modern / GTM 中已经重叠的材料。
- `docs/gcy-material-id-map.json` 中记录的 old id -> GTM id 差异。
- 后续发现的其它重复材料 id 差异。

决策：

所有重叠材料的 canonical id 统一使用 GTM / GTCEu Modern id。GCYM 不为了保留
GCY 原版旧 id 重新注册同义材料。

迁移规则：

- `docs/gcy-material-id-map.json` 只作为旧配方迁移输入表：左侧 old id 用来识别 GCY
  原版引用，右侧 `modern_id` 是迁移后的实际 id。
- 旧 id 不作为 runtime alias 或新 material id 保留。
- 配方、tag、材料引用和文档中的新实现均以 GTM id / field 为准。

## 0.1. 电压等级命名变化

差异：

- GCY 原版 `GAValues` 中 index `12` 是 `UMV`，电压值为 `134217728`。
- GTCEu Modern / GTM `GTValues` 中 index `12` 是 `UXV`，电压值同为 `134217728`。
- GCY 原版 index `13` 是 `UXV`，电压值为 `536870912`。
- GTM index `13` 是 `OpV`，电压值同为 `536870912`。
- GCY `MAX` 使用 `Integer.MAX_VALUE`；GTM `MAX` 使用 `2147483648L`。涉及
  `MAX` 的配方或机器参数需要单独检查，不按名称盲替。

决策：

迁移时按 GTM / GTCEu Modern 的 tier 命名和数值标准实现。GCYM 不恢复 GCY 原版
`UMV` tier 名称，也不把 GCY 的 `UXV` 作为 GTM `UXV` 使用。

迁移规则：

- 原版 `GAValues.UMV` / `GAValues.V[GAValues.UMV]` 迁移为 GTM
  `GTValues.UXV` / `GTValues.V[GTValues.UXV]`。
- 原版 `GAValues.UXV` / `GAValues.V[GAValues.UXV]` 迁移为 GTM
  `GTValues.OpV` / `GTValues.V[GTValues.OpV]`。
- 原版 `GAValues.MAX` / `GAValues.V[GAValues.MAX]` 只在具体引用点单独处理；
  若配方实际使用的是 adjusted EU/t，应优先检查是否应迁移到 `GTValues.VA`。
- 带电压等级语义的材料名、marker material、cable property、recipe tier、machine tier
  都按 GTM tier 标准迁移。原版 `UMV*` 超导材料迁移为现代 `UXV*`，原版 `UXV*`
  超导材料迁移为现代 `OpV*`；旧 id 只保留在 id map 中作为迁移输入键。

分包：

- `voltage/superconductors`：带 tier 名称的超导材料及 base 材料，例如旧
  `umv_superconductor(_base)` -> 现代 `uxv_superconductor(_base)`，旧
  `uxv_superconductor(_base)` -> 现代 `opv_superconductor(_base)`。
- `voltage/tier_markers`：旧 `UMV` / `UXV` marker material 到现代 `UXV` / `OpV`
  marker 的迁移映射。仅用于 circuit、tier marker 等 recipe 输入语义；不恢复旧
  `UMV` marker 作为 canonical material。
- 后续源码注册时按这个分包拆模块；不要因为旧构造器是 `IngotMaterial` 或
  `MarkerMaterial` 就放回通用 ingot/marker 包。

## 1. GTM Element-Only, GCY Old Ingot Materials

适用范围：

`Zirconium`, `Polonium`, `Copernicium`, `Francium`, `Radium`, `Actinium`,
`Hafnium`, `Rhenium`, `Technetium`, `Thallium`, `Germanium`, `Selenium`,
`Astatine`, `Rutherfordium`, `Dubnium`, `Seaborgium`, `Bohrium`, `Tennessine`,
`Livermorium`, `Moscovium`, `Nihonium`, `Roentgenium`, `Meitnerium`.

决策：

这些材料不作为新材料重新注册，也不因为 GTCEu Modern 已有同名 element
就直接跳过。实际处理方式是在 GCYM/addon 侧复用 GTM 已有 material，并为它们补回
GCY 原版中应该存在的物态。

当前选择的最低处理：

- 对上述材料按实际配方需求补回 GCY 原版中需要的物态；不在材料去重阶段无条件补全
  所有 old `IngotMaterial` 自动表面。
- 以 GTM 的 material id / element binding 为 canonical；`docs/gcy-material-id-map.json`
  只用于把原版旧 id 迁移到 GTM id，不恢复原版旧 id。
- 不把这些材料从 GTM duplicate list 中移除；它们仍然是 GTM 已有材料，只是需要
  addon 后处理补必要的 property/form。

额外注意：

- `Selenium` 原版字段声明是 `DustMaterial`，但构造器实际是 `new IngotMaterial(...)`；
  这里按构造器行为处理为 old ingot case。
- `Polonium`、`Copernicium`、`Radium`、`Technetium` 不在材料去重阶段预先补
  ingot 或 fluid。已知 GCY 显式配方中没有看到这四者的 `ingot` 输入/输出；
  `Polonium`、`Copernicium`、`Radium` 有显式 fluid 用法，`Technetium` 目前只看到
  dust 用法。具体需要补哪些形态，等迁移到对应配方时再按实际引用处理。
- `Thallium` 原版 id 是 `thalium`，GTM canonical id 是 `thallium`。后续配方引用
  旧 id 时走 id map/remap，不额外注册 `thalium` 材料。
- 超重元素虽然在本轮差异表里按非核 duplicate 处理，但如果后续核边界重新收紧，
  应优先按“核相关内容搁置”的规则暂停实现。

实现约束：

- 只做 material property/form 补充，不复制原版整套颜色、icon 或全部生成 flags。
- 不覆盖 GTM 已经定义的 element、color、secondaryColor、radioactive icon 等基础定义。
- 如果 GTM 已经有更完整的特殊性质，默认保留 GTM 行为；只补 GCY 配方会依赖的缺失物态。

待验证：

- 配方迁移时检查是否有 `ingot/dust/fluid` 形式引用这些材料。
- 检查补 form 后是否触发多余 autogenerated recipes。
- 若自动生成内容过多，再针对单个材料追加禁用 flags，而不是扩大到整组。

## 2. `AmmoniumChloride`

差异：

- GCY 原版把 `AmmoniumChloride` 注册为 `FluidMaterial`，并在配方里使用
  `AmmoniumChloride.getFluid(...)`。
- GTCEu Modern 把 `AmmoniumChloride` 注册为 `dust()`，components 为
  `Ammonia + HydrochloricAcid`，并通过 dust 参与配方。
- GTM 中未看到“`AmmoniumChloride` 粉末 + 水 -> 同名水溶液”的现成转换模式。

决策：

使用 GTM 现有 `AmmoniumChloride` 粉末作为 canonical material。暂不为
`AmmoniumChloride` 补普通 fluid，也不新建 `AmmoniumChlorideSolution`。
不修改 GTM 现有 compound/components；保持 GTM 的
`Ammonia + HydrochloricAcid` 组分定义。

配方迁移变化：

- 原版所有 `fluidInputs(AmmoniumChloride.getFluid(...))` 需要改写为
  `inputItems(dust, AmmoniumChloride, ...)`。
- 原版所有 `fluidOutputs(AmmoniumChloride.getFluid(...))` 需要改写为
  `outputItems(dust, AmmoniumChloride, ...)`。
- 数量不能机械按 `1000 mB = 1 dust` 自动套用；需要按每条反应式重新平衡。
  初始迁移可用 `1000 mB -> 2 dust` 作为 GTM 参照，因为 GTM 现有
  `sodium_bicarbonate_from_salt` 输出 `2x AmmoniumChloride dust`，且
  decomposition handler 对该 dust 的化学式按 `2 dust -> NH3 + HCl` 处理。

已知受影响的原版链条：

- `OpticalChain`: 生成/消耗 `NH4Cl` 的 vanadate 相关步骤。
- `VanadiumChain`: `NH4Cl` 参与 `AmmoniumVanadate` 生成。
- `ZirconChain`: zircon/hafnium 分离中消耗 `NH4Cl`，以及 `NH4Cl -> NH3 + HCl`
  的分解步骤。
- `FullereneChain`: `NH4Cl` 副产与分解回 `HCl + NH3`。
- `PlatinumSludgeGroupChain`: iridium / platinum sludge 处理中消耗或生成
  `NH4Cl`。
- `SuperconductorsSMDChain`, `HNIWChain`, `Batteries`: 多处将 `NH4Cl` 作为
  副产物输出。

实现约束：

- 不把这些配方迁移成 `molten_ammonium_chloride`；这里不是熔融物。
- 不把 `HydrochloricAcid` 改成 `DilutedHydrochloricAcid`，除非具体配方显式要
  水溶/稀酸语义。
- 若某条原版配方因为流体槽数量改成 dust 后更适合小化工机，仍先保持原 recipe type，
  后续按链条平衡再调整机器类型。

## 3. `AquaRegia`

差异：

- GCY 原版把 `AquaRegia` 注册为 `NitricAcid:1 + HydrochloricAcid:1` 的
  `FluidMaterial`。
- GTCEu Modern 已有 canonical `AquaRegia` liquid，components 为
  `NitricAcid:1 + HydrochloricAcid:2`。
- GCY 原版有 `NitricAcid 1000 mB + HydrochloricAcid 1000 mB -> AquaRegia 2000 mB`
  的 mixer 配方；GTM 现有配方是
  `NitricAcid 1000 mB + HydrochloricAcid 2000 mB -> AquaRegia 3000 mB`。

决策：

使用 GTM 现有 `AquaRegia` material 和 GTM 现有生产配方。GCYM 不重新注册
`AquaRegia`，不覆盖 GTM 的 `1:2` 酸配比，也不保留 GCY 原版 `1:1` 生产配方。

配方迁移变化：

- 原版消耗 `AquaRegia.getFluid(...)` 的配方继续引用 GTM `AquaRegia`。
- 若某条 GCY recipe 的化学注释按 `1:1` aqua regia 书写，迁移时以 GTM
  `AquaRegia` 的实际 fluid 数量为准，不反推改动 GTM compound。

## 4. `Polybenzimidazole`

差异：

- GCY 原版把 `Polybenzimidazole` 注册为 old `IngotMaterial`，通过
  `SMELT_INTO_FLUID`、`GENERATE_FOIL` 和 fluid pipe property 同时提供固体统一项、
  foil、液体和流体管相关用途。
- GTCEu Modern 已有 canonical `Polybenzimidazole`，注册为
  `.polymer().liquid(...)`，并已经提供 foil、plate/ingot 相关生成用途、普通
  liquid 和 fluid pipe property。
- GTM 已有 PBI 生产链，但 GCY 原版还存在一条 Chemical Plant 路线。

决策：

使用 GTM 现有 `Polybenzimidazole` 作为 canonical material。GCYM 不重新注册
`polybenzimidazole`，也不新增替代 id。

生产配方处理：

- 保留 GTM 原有 `polybenzimidazole` 生产配方，不覆盖、不删除、不按 GCY 副产物改写。
- 额外加入 GCY 原版 Chemical Plant 路线：
  `Oxygen + Ethylene + Ammonia + NitricAcid + Benzene -> Polybenzimidazole + Water`。
- GCY 原版 `PolymerChain` 中与 GTM 同入口的
  `Diaminobenzidine + Diphenylisophtalate -> Polybenzimidazole` 不迁移为覆盖配方；
  后续若需要保留 GCY 副产物差异，必须另行确认是否作为可并存 recipe 添加。

配方迁移变化：

- 原版所有 `Polybenzimidazole.getFluid(...)` 改为引用 GTM
  `Polybenzimidazole` liquid。
- 原版所有 `foil/plate/ingot/pipeSmall` 等 PBI 统一项改为引用 GTM
  `Polybenzimidazole` 对应 tag prefix。
- GCY 独特用途继续保留，例如 cable insulation 阶梯、Bioware/Optical/Wetware
  组件、PEEK 链前置 catalyst、GCY mega machines、Laser-Safe Casing、
  `SEPARATION_ELECTROMAGNET`、`NANOTOME` 和 field shape molds。

实现约束：

- 不修改 GTM PBI 的 material property、温度、颜色、toolStats 或 fluid pipe
  参数，除非后续配方迁移发现缺失物态。
- 不把 PBI liquid 当作 molten 处理；这里使用 GTM material 的普通 liquid。
- 若某个 GCY recipe 依赖 `Polybenzimidazole` 的旧管道吞吐或温度数值，先记录差异，
  不在本轮覆盖 GTM material property。

## 5. `PotassiumHydroxide`

差异：

- GCY 原版把 `PotassiumHydroxide` 注册为 `SimpleFluidMaterial`，配方中使用
  `PotassiumHydroxide.getFluid(...)`。
- GTCEu Modern 把 `PotassiumHydroxide` 注册为 `dust(1)`，components 为
  `Potassium + Oxygen + Hydrogen`，并带皮肤接触化学灼伤 hazard。
- GTM 自带配方只看到 dust 用法，未看到 KOH liquid 或 KOH solution。

决策：

使用 GTM 现有 `PotassiumHydroxide` 粉末作为 canonical material。GCYM 不补
`PotassiumHydroxide` liquid，也不新建 `PotassiumHydroxideSolution`。

配方迁移变化：

- 原版所有 `fluidInputs(PotassiumHydroxide.getFluid(...))` 改写为
  `inputItems(dust, PotassiumHydroxide, ...)`。
- 原版所有 `fluidOutputs(PotassiumHydroxide.getFluid(...))` 改写为
  `outputItems(dust, PotassiumHydroxide, ...)`。
- 数量按 GTM 现有 KOH 配方参照：`1000 mB KOH -> 3 dust PotassiumHydroxide`。
  例如 `2000 mB -> 6 dust`、`4000 mB -> 12 dust`、`6000 mB -> 18 dust`。
- 原版少量流体催化剂用法也改为整份 dust 催化剂：
  `notConsumable(PotassiumHydroxide.getFluid(1))` 和
  `notConsumable(PotassiumHydroxide.getFluid(10))` 迁移为
  `notConsumable(dust, PotassiumHydroxide)`。

催化剂口径：

- GTM 和 GCY 现有 `notConsumable` 粉末催化剂大多使用整份 `dust`，没有看到
  `notConsumable(dustSmall, ...)` 或 `notConsumable(dustTiny, ...)` 的常规模式。
- 这里选择整份 `dust`，保持 GTM / GCY 既有粉末催化剂写法一致；不为 KOH 引入
  small/tiny dust 的特殊规则。
- 这仍然是行为变化：整份 dust 是物品催化剂，不再是流体槽中的微量 KOH。

已知受影响的原版链条：

- `QuantumDotsChain`: soap 生产消耗 KOH。
- `SuperconductorsSMDChain`, `OpticalChain`: manganate / permanganate 循环中消耗或生成 KOH。
- `FullereneChain`: `KCN` 生成、`RockSalt + Water -> KOH + HCl`、benzoyl chloride
  相关步骤。
- `SensorEmitter`, `NiobiumTantalumChain`: tantalum / rhenium / technetium 相关步骤。
- `Batteries`: nickel oxyhydroxide 生产和 NiMH 电池填充。
- `NaquadahChain`, `CombinedChains`: peroxymonosulfate / naquadah compound 相关步骤。
- `Dyes`: ferrocyanide、indanthrone blue、mauveine、indigo 等步骤，其中
  indanthrone blue 和 indigo 使用整份 dust 催化剂。
- `UraniumChain`: 涉及 KOH 的核相关步骤继续搁置。

## 6. `TantalumCarbide`

差异：

- GCY 原版把 `TantalumCarbide` 注册为 `SimpleDustMaterial`，配方中只按 dust-like
  item stack 使用。
- GTCEu Modern 已有 canonical `TantalumCarbide`，注册为
  `.ingot(4).fluid()`，并提供 plate / double plate 等更完整材料面。
- GTM 现代 GCYM 已有 `Tantalum + Carbon -> TantalumCarbide dust` 的 mixer 配方。

决策：

使用 GTM 现有 `TantalumCarbide` 作为 canonical material。GCYM 不重新注册
GCY 原版 simple dust。

配方迁移变化：

- 原版 `TantalumCarbide.getItemStack(...)` 改写为
  `inputItems/outputItems(dust, TantalumCarbide, ...)`。
- 不覆盖 GTM 现有 `TantalumCarbide` 生产路线。
- GCY 原版 `Tantalum + Graphite -> TantalumCarbide` blast 路线默认不迁移；若后续需要
  保留为额外路线，必须单独确认。

## 7. `Xenon`, `Neon`, `Krypton`

差异：

- GCY 原版把这三个元素注册为 old `FluidMaterial`。
- GCY 原版 `Xenon` 带 `GENERATE_FLUID_BLOCK`，但不带 `GENERATE_PLASMA`。
- GCY 原版 `Neon` 和 `Krypton` 带 `GENERATE_PLASMA`。
- GTCEu Modern 已有 canonical `Xenon`、`Neon`、`Krypton`，但都只注册为
  `.gas()`。
- GCY 原版另有独立 `SimpleFluidMaterial` `LiquidXenon`，以及
  `KryptonDifluoride`、`XenonTrioxide`、`NobleGases` 等派生/混合流体材料。

决策：

使用 GTM 现有 `Xenon`、`Neon`、`Krypton` 作为 canonical element materials。
GCYM 不重新注册同 id old `FluidMaterial`。

在 addon 侧为三个 GTM 气体补充额外 fluid storage keys：

- 为 `Xenon`、`Neon`、`Krypton` 补 `LIQUID`。
- 为 `Xenon`、`Neon`、`Krypton` 补 `PLASMA`。
- 补充后仍保持 `GAS` 为 primary key，保证 `material.getFluid(...)` / recipe 中
  默认流体语义继续指向气态。

配方迁移变化：

- 原版 `Xenon.getFluid(...)`、`Neon.getFluid(...)`、`Krypton.getFluid(...)`
  继续迁移为对应 GTM material 的 gas。
- 原版 `Neon.getPlasma(...)`、`Krypton.getPlasma(...)` 迁移为对应 GTM material 的
  `PLASMA`。
- 原版独立 `LiquidXenon.getFluid(...)` 迁移为 GTM `Xenon` 的 `LIQUID`，不再单独注册
  `liquid_xenon` material。
- 若后续出现或新增 `Xenon.getPlasma(...)` 用法，也使用 GTM `Xenon` 的 `PLASMA`。

实现约束：

- 不把 `LIQUID` 设置成 primary key；避免破坏 GTM 现有气体配方。
- `LiquidXenon` 是迁移 alias，不作为新 material 保留。
- `KryptonDifluoride`、`XenonTrioxide`、`NobleGases` 是独立派生/混合流体材料，
  不因为本节决策自动并入元素本体。
- `Xenon` 的 `PLASMA` 是本轮新增的统一补充；GCY 原版本体没有显式
  `GENERATE_PLASMA`。

## 8. `RhodiumSulfate`

差异：

- GCY 原版把 `RhodiumSulfate` 注册为 `FluidMaterial`，components 写作
  `Rhodium + Sulfur + Oxygen + RareEarth`。
- GCY 原版还注册了独立的 `RhodiumSulfateSolution`，用于铂族泥中较长的铑处理链。
- GTCEu Modern 已有 canonical `RhodiumSulfate` liquid，components 为
  `Rhodium:2 + Sulfur:3 + Oxygen:12`，formula 为 `Rh2(SO4)3`。
- GTM 未看到 `RhodiumSulfateSolution`。

决策：

材料层面使用 GTM 现有 `RhodiumSulfate`。GCYM 不重新注册 GCY 原版
`RhodiumSulfate`，不修改 GTM 的 compound/formula，也暂不新增
`RhodiumSulfateSolution`。

配方迁移变化：

- 原版涉及 `RhodiumSulfate` / `RhodiumSulfateSolution` 的铂族泥处理配方暂不在
  材料去重阶段处理。
- 这部分等具体设计铂族泥产线时再统一决定：可以选择接入 GTM 短线，也可以单独评估是否保留
  GCY 的 solution / crude rhodium / rhodium salt / nitrate / filter cake 长线。

## 9. 剩余 `Components Differ` 材料

范围：

- `QuartzSand`
- `IrMetalResidue`
- `Pyrochlore`
- `Potin`
- `IncoloyMA956`
- `Zeron100`

差异：

- 这些材料在 GCY 原版和 GTCEu Modern 中都有 components，但组分定义不同。
- 前面已单独处理的 `AquaRegia`、`AmmoniumChloride`、`RhodiumSulfate` 不再重复列入本节。
- `Pyrochlore` 的原版 components 包含核相关元素；核相关产线仍按此前决定继续搁置。

决策：

材料层面统一使用 GTM 现有 canonical material。GCYM 不重新注册这些 GCY 原版同名材料，
也不修改 GTM 的 compound/components/formula。

配方迁移变化：

- 材料去重阶段只处理引用映射，不提前重平衡这些材料的生产线。
- 若后续具体设计相关产线时发现 GCY 配方需要保留，按单条链路评估是否接入 GTM 材料，
  不反向改变材料注册定义。

## 10. `Chrome` -> `Chromium` 命名变化

范围：

- `ChromiumTrioxide`
- `PotassiumDichromate`
- 其他从 GCY 原版迁移来的 components 或配方代码中直接引用 old GTCE `Chrome` 的位置。

差异：

- GCY 原版沿用 old GTCE 的元素名 `Chrome`。
- GTCEu Modern 中对应元素名为 `Chromium`。
- 已确认的重复材料差异中，`ChromiumTrioxide` 和 `PotassiumDichromate` 只是 components
  里使用 `Chrome` / `Chromium` 的命名差异。

决策：

迁移到 GCYM 时统一使用 GTM / GTCEu Modern 的 `Chromium` 命名。这个变化只作为
API / symbol 命名迁移记录，不作为独立材料重注册理由。

配方迁移变化：

- 原版 `Chrome` material 引用迁移为 GTM `Chromium`。
- 原版 `ChromiumTrioxide`、`PotassiumDichromate` 使用 GTM 现有 canonical material，
  不因 `Chrome` / `Chromium` 名称差异重新注册。
- 若旧注释或化学式中写作 `Chrome`，代码实现以 `Chromium` 为准；注释可在迁移时顺手改成
  `Chromium`，但不单独改变配方平衡。

## 11. Formula-only / Display-formula-only 差异

范围：

- `NaquadriaSolution`
- `RubySlurry`
- `SapphireSlurry`
- `GreenSapphireSlurry`
- `Agar`
- `Diaminobenzidine`
- `Dichlorobenzidine`

差异：

- 这些重复材料的差异只体现在 formula 或 display formula 文本层面。
- `Diaminobenzidine` 和 `Dichlorobenzidine` 属于 GTM 侧已有 formula，而 GCY 原版没有等价
  formula 信息。

决策：

全部使用 GTM 现有 canonical material。GCYM 不为了保留 GCY 原版 formula /
display formula 文本重新注册这些材料，也不覆盖 GTM 的 formula。

配方迁移变化：

- 材料引用直接映射到 GTM material。
- 旧 formula/display formula 只作为历史差异记录，不参与配方平衡和材料注册设计。
